# Project Overview

This project implements a **secure workflow verification framework** designed for embedded systems.

The goal is to verify that a device executes a program **according to the original workflow**, preventing manipulation of sensor data or execution logic.

The system verifies that the **execution trace generated by the device is compatible with the workflow extracted from the original program source code**.

---

# Device Runtime (C)

The embedded runtime implements secure computation primitives and execution tracing.

Main modules include:

### Crypto Module
Provides encryption primitives based on **AES using mbedTLS**.

### int_var Module
Implements a secure integer abstraction used by the runtime.  
Values are stored in encrypted form.

### Trace Module
Records the sequence of API calls executed by the device.

### API Module
Defines the secure operations that can be executed by the program.

Examples include:

- initialization
- assignment
- arithmetic operations
- sensor data acquisition

Each API invocation is recorded inside the execution trace.

---

# Workflow Extractor (Python)

The extractor analyzes a C program and reconstructs the workflow of API calls.

The extractor:

1. parses the source code
2. detects API calls
3. reconstructs the execution workflow
4. generates the workflow graph

This graph is later used by the verifier.

---

# Workflow Verifier (Java)

The verifier validates execution traces received from the device.

The verification process checks whether the sequence of API calls executed by the device is compatible with the workflow graph.

The workflow graph is implemented using an **adjacency matrix representation**.

The verification algorithm checks:


trace ∈ workflow_graph


If all transitions between APIs are valid → execution is accepted.

Otherwise → execution is rejected.
