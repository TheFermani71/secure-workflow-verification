## Project Overview

This project implements a **secure workflow verification framework** for embedded systems.

The goal is to ensure that a device executes a program **according to the original workflow**, preventing manipulation of sensor data or program logic.

The system verifies that the **execution trace generated by the device is compatible with the workflow extracted from the original source code**.

This approach allows detecting:

- program tampering
- unexpected execution paths
- manipulation of computation results
- incorrect API invocation sequences

---

# Implemented Modules

## Device Runtime

The device runtime implements secure computation primitives.

Modules include:

### Crypto Module
Provides encryption primitives based on **AES using mbedTLS**.

### int_var Module
Implements a **secure integer abstraction** used by the runtime.  
Values are stored in encrypted form to protect computation integrity.

### Trace Module
Records the sequence of API calls executed by the device.

The trace is later used for execution verification.

### API Module
Defines the secure operations available to the program.

Examples include:

- initialization
- assignment
- arithmetic operations
- data acquisition

Each API invocation is recorded in the execution trace.

---

## Verifier Module

The verifier validates the execution trace received from the device.

It checks whether the sequence of executed APIs is compatible with the **workflow graph extracted from the original program**.

The workflow is represented internally as an **adjacency matrix**.

The verifier algorithm checks:


trace ∈ workflow_graph


If all transitions are valid → execution is accepted.

Otherwise → execution is rejected.

---

## Extractor Module

The extractor analyzes the C source code and reconstructs the **workflow graph**.

The extractor:

1. parses the source file
2. detects API calls
3. reconstructs the sequence of operations
4. generates the workflow graph

This graph is used by the verifier to validate execution traces.
