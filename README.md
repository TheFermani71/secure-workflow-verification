# 1. Project Overview

## 1.1 Introduction

This project implements a **secure workflow verification architecture for embedded systems**.

The goal of the system is to verify that a device executes **only the operations allowed by the program workflow**, even if the device software could potentially be manipulated.

The system records the sequence of executed API calls (**execution trace**) and verifies that it is compatible with the **workflow graph extracted from the original program source code**.

---

## 1.2 Motivation

Embedded systems are frequently exposed to risks such as:

- firmware manipulation
- unexpected execution paths
- incorrect API invocations
- manipulation of computation results

The proposed architecture allows detecting such anomalies by verifying that the execution trace follows the **expected workflow model extracted from the program**.

---

# 2. Formal Verification Model

## 2.1 Workflow Verification Problem

The execution verification problem can be formally expressed as:


trace ∈ L(G)


Where:

- **trace** = sequence of API calls executed by the device  
- **G** = workflow graph extracted from the program source code  

If the trace belongs to the language generated by the workflow graph, the execution is considered **valid**.

Otherwise, the system detects a **workflow violation**.

---

## 2.2 Trace Representation

An execution trace represents the **ordered sequence of API calls executed by the device**.

Example trace:


INIT
ASSIGN
ASSIGN
ADD
DIV


Each element of the trace corresponds to an API invocation performed during program execution.

# 6. Secure Runtime Modules

## 6.1 Crypto Module

The crypto module provides **authenticated encryption** using AES-GCM through the **mbedTLS library**.

Main features include:

- secure device key storage
- random IV generation
- authenticated encryption
- integrity protection

---

## 6.2 int_var Module

The `int_var` module implements a **secure integer abstraction**.

Each variable internally stores:

- value
- timestamp
- variable identifier

The internal structure is serialized and **encrypted before storage**.

---

## 6.3 Trace Module

The trace module records API executions.

Each trace entry contains:


api_id
var1
var2
result
timestamp


The trace can be exported and used for verification.

# 7. Workflow Extraction

## 7.1 Extractor Overview

The Python extractor analyzes the C program and reconstructs the sequence of API calls.

The extractor performs the following operations:

1. parse the source code
2. detect API calls
3. reconstruct the execution sequence
4. generate workflow transitions
5. export the workflow graph

---

## 7.2 Example Program

Example program analyzed by the extractor:


api_init(a)
api_assign(a,10)
api_assign(b,5)
api_add(a,b,result)
api_div(result,b,result)


Extracted workflow:


INIT → ASSIGN → ADD → DIV


---

## 7.3 Generated Workflow Graph

The workflow graph is exported as a JSON file.

Example:

```json
{
  "edges": [
    [1,2],
    [2,3],
    [3,4]
  ]
}

# 8. Workflow Verification

## 8.1 Verifier Overview

The Java verifier validates execution traces against the workflow graph extracted from the program source code.

The verifier loads the workflow graph generated by the Python extractor and verifies whether the sequence of API calls executed by the device follows the allowed transitions defined in the workflow.

The verification rule is:


∀ transitions (prev → next) in trace
must exist in workflow graph


If a transition between two consecutive API calls is not present in the workflow graph, the execution is considered invalid.

---

## 8.2 Verification Outcome

If the execution trace respects all transitions defined in the workflow graph:


TRACE VALID


Otherwise, if at least one transition violates the workflow model:


TRACE INVALID


---

# 9. Trace Parsing

## 9.1 Trace File Format

Execution traces are stored in text files generated by the device.

Example trace file:


TRACE
INIT
ASSIGN
ADD
DIV
END


The keywords `TRACE` and `END` act as delimiters and are ignored by the parser.

---

## 9.2 API Identifier Mapping

The TraceParser converts API names into numeric identifiers used by the verifier.

Mapping used by the system:


INIT → 1
ASSIGN → 2
ADD → 3
DIV → 4


---

## 9.3 ExecutionTrace Representation

After parsing the trace file, the execution sequence becomes:


[1,2,3,4]


This representation is stored inside the `ExecutionTrace` object and used by the verification algorithm.
